<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SDK client -> UploadKit API | API key transmitted in Authorization header over HTTPS |
| SDK client -> R2 presigned URL | File bytes sent directly to R2; URL is time-limited |
| SDK consumer -> SDK | Developer provides apiKey; SDK stores in memory only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | http.ts | mitigate | API key sent only in Authorization header, never in URL query params or logs. fetchApi never logs the apiKey value. |
| T-04-02 | Spoofing | http.ts | mitigate | All API calls use HTTPS (baseUrl defaults to https://). fetchApi rejects http:// baseUrl. |
| T-04-03 | Tampering | single.ts / multipart.ts | accept | Presigned URLs are time-limited (15min) and content-type locked. SDK sends matching Content-Type. Server validates on complete. |
| T-04-04 | Denial of Service | retry.ts | mitigate | Max retry count capped (default 3), exponential backoff with jitter prevents thundering herd. AbortController available. |
| T-04-05 | Information Disclosure | client.ts | mitigate | UploadKitClient stores apiKey in a private field, not enumerable. No toString/toJSON that leaks the key. |
| T-04-06 | Elevation of Privilege | validation.ts | accept | Client-side validation is UX convenience only. Server enforces all limits authoritatively. |
</threat_model>
