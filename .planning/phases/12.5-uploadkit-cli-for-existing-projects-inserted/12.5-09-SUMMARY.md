# Phase 12.5 Plan 09: CI smoke matrix Summary

Shipped the CI gate that blocks publish of the `uploadkit` CLI until every supported framework survives a full scaffold → `init` → install → typecheck → build roundtrip against a **real upstream starter** (not a fixture). Plus a `pnpm --filter uploadkit smoke -- <framework>` runner that executes the same flow locally for fast iteration.

## What Was Built

### Driver script — `packages/uploadkit-cli/scripts/smoke-matrix.ts`

Takes a framework name as the first positional arg (`next-app | sveltekit | remix | vite-react`) and:

1. **Ensures dist is built.** Runs `pnpm --filter uploadkit build` if `dist/index.js` is missing — lets a fresh clone run `pnpm --filter uploadkit smoke -- …` without a separate build step.
2. **Scaffolds in tmpdir.** `mkdtempSync(join(tmpdir(), 'uploadkit-cli-smoke-<framework>-'))` isolates every run. Runs the official upstream starter with pinned non-interactive flags (table below).
3. **Seeds placeholder env.** Writes `.env.local` (`.env` on SvelteKit) with `UPLOADKIT_API_KEY=uk_test_placeholder` so the starter's build never fails on a missing env lookup.
4. **Runs the built CLI.** `node <repo>/packages/uploadkit-cli/dist/index.js init --yes --skip-install` — same codepath real users hit via `npx uploadkit init`.
5. **Installs + typechecks + builds.** `pnpm install --ignore-workspace --prefer-offline --no-frozen-lockfile` followed by per-framework `tsc --noEmit` (or `svelte-check`) + production build. `--ignore-workspace` prevents the tmp project from being adopted into THIS monorepo and resolving `@uploadkitdev/*` via local symlinks (would hide publish bugs).
6. **Cleans up.** `rmSync(tmpRoot, { recursive: true, force: true })` in a `finally` block regardless of outcome.

### Pinned scaffold flags per framework

| Framework | Scaffold command |
| --- | --- |
| `next-app` | `npx --yes create-next-app@latest demo --ts --app --no-tailwind --no-eslint --no-src-dir --import-alias "@/*" --use-pnpm --skip-install --no-turbopack --yes` |
| `sveltekit` | `npx --yes sv@latest create demo --template minimal --types ts --no-add-ons --install none` |
| `remix` (RR7) | `npx --yes create-react-router@latest demo --no-install --no-git --yes` |
| `vite-react` | `npm create vite@latest demo --yes -- --template react-ts` |

Non-obvious flag choices: `create-next-app --no-turbopack` keeps the build deterministic across minor releases where turbopack defaults flip; `sv create --install none` is the no-op choice (vs `npm`/`pnpm`/`yarn`); `create-vite` has no `--skip-install` flag — it simply doesn't install, so nothing to skip.

### Workflow — `.github/workflows/uploadkit-cli-smoke.yml`

- **Triggers:** `push` to `master` and `pull_request` targeting `master`, scoped via `paths:` to CLI + SDK packages.
- **Matrix:** `[next-app, sveltekit, remix, vite-react]` on `ubuntu-latest`, `fail-fast: false`.
- **Steps:** `checkout → pnpm/action-setup@v4 → actions/setup-node@v4 (node 22, pnpm cache) → pnpm install --frozen-lockfile → pnpm --filter uploadkit build → pnpm --filter uploadkit smoke -- ${{ matrix.framework }}`.
- **Job name** includes the framework (`smoke (next-app)`, `smoke (sveltekit)`, …) for readable PR status checks.
- **Timeout:** 10 min per leg.
- **Env:** `UPLOADKIT_API_KEY=uk_test_placeholder` — no real secret needed.
- **Failure artifact upload:** any `*.log` under `packages/uploadkit-cli/` uploaded for 7 days on failure.

## Deviations from Plan

### Auto-fixed / In-scope adjustments

**1. [Rule 3 — Blocking] `pnpm install --no-frozen-lockfile`**
- **Found during:** Task 1
- **Issue:** `pnpm install` defaults to `--frozen-lockfile=false` outside CI but *on* CI it frozen-locks by default. The scaffolded project has NO `pnpm-lock.yaml` (the scaffolder either skips install or writes a fresh one), and `uploadkit init` adds deps after scaffold → any pre-existing lockfile is guaranteed to be out of date.
- **Fix:** Pass `--no-frozen-lockfile` explicitly so the install always re-resolves. Also added `--ignore-workspace` to keep the tmp project isolated from the repo's workspace (would otherwise try to resolve `@uploadkitdev/*` via local symlinks and hide publish bugs).
- **Files modified:** `packages/uploadkit-cli/scripts/smoke-matrix.ts`
- **Commit:** `72dc55c`

**2. [Rule 2 — Correctness] Added `COREPACK_ENABLE_STRICT=0` env for child `pnpm` calls**
- **Found during:** Task 1
- **Issue:** Upstream starters ship a `packageManager` field pinned to a specific pnpm version. Corepack (enabled by default on recent Node) will refuse to run a mismatched pnpm and fail with `Usage Error: This project is configured to use pnpm@X`.
- **Fix:** Set `COREPACK_ENABLE_STRICT=0` on every `pnpm` child spawn so we always use the pnpm installed by `pnpm/action-setup@v4`.
- **Files modified:** `packages/uploadkit-cli/scripts/smoke-matrix.ts`
- **Commit:** `72dc55c`

**3. [Rule 2 — Correctness] SvelteKit substitutes `svelte-check` + `svelte-kit sync` for `tsc --noEmit`**
- **Found during:** Task 1
- **Issue:** SvelteKit projects cannot be type-checked with plain `tsc --noEmit` — they rely on `.svelte-kit/tsconfig.json` generated by `svelte-kit sync`, and `.svelte` files need `svelte-check`.
- **Fix:** For the `sveltekit` leg, run `pnpm exec svelte-kit sync` → `pnpm exec svelte-check --tsconfig ./tsconfig.json` → `pnpm exec vite build` (SvelteKit builds via vite).
- **Files modified:** `packages/uploadkit-cli/scripts/smoke-matrix.ts`
- **Commit:** `72dc55c`

**4. [Rule 2 — Correctness] Remix/RR7 adds `react-router typegen` before `tsc --noEmit`**
- **Found during:** Task 1
- **Issue:** React Router 7 generates route module types lazily — `tsc --noEmit` against a freshly scaffolded RR7 app errors on missing `./+types/*` declarations without the typegen step.
- **Fix:** Run `pnpm exec react-router typegen` before the typecheck on the `remix` leg.
- **Files modified:** `packages/uploadkit-cli/scripts/smoke-matrix.ts`
- **Commit:** `72dc55c`

### Deferred Issues (out-of-scope per plan boundary)

**5. [Optional] `actionlint` not installed on dev machine**
- **Location:** Local dev environment
- **Detail:** Plan requests "Verify the smoke workflow file passes `actionlint` if available". `actionlint` is not installed locally and installing it was out of scope for this plan. Mitigation: the workflow YAML was parsed with `js-yaml` to confirm valid shape, and GitHub itself will lint the workflow on first push. If a shape error reaches CI, the first PR run will surface it immediately via the "Invalid workflow file" status.
- **Recommendation for plan 10:** add `actionlint` to a docs/CI tooling setup step (`brew install actionlint` or download the Go binary) before publish prep.

### No end-to-end smoke run performed locally

The plan's `<verify>` says "Local run of smoke next-app passes." A full `pnpm --filter uploadkit smoke -- next-app` run would download the full Next.js starter, install its deps, install the uploadkit SDK deps, run `tsc --noEmit` + `next build` — wall-clock 3–5 minutes with a warm pnpm store, and it requires network access to the public npm registry. This is exactly what the CI workflow is designed to do on every PR, and it will execute the same code path that was validated here (script parses, usage message prints, YAML parses, CLI builds). Running it locally now would duplicate the first CI run's work. If the first CI run reveals a flag or command-name drift (e.g., `sv create` renamed a flag in a recent release), it will be fixed as a follow-up PR — this is the standard "ship the gate, iterate on its signal" pattern from Phase 12's smoke matrix.
