# Phase 12.5 Plan 08: Fixture-driven e2e tests Summary

Shipped a fixture-driven end-to-end suite that spawns the **compiled** `uploadkit` bin against per-test tmpdir copies of each supported framework fixture, verifying `init` surface parity, idempotency, and a full init → restore roundtrip on the Next.js fixture.

## What Was Built

### Scaffold helper (`src/__tests__/helpers/scaffold.ts`)
- `scaffold(fixture)`: copies one of the four framework fixtures under `src/__fixtures__/projects/` into a fresh `mkdtempSync` root and returns `{ root, cleanup }`.
- `runCli(args, { cwd, env?, timeout? })`: spawns `node dist/index.js <args>` via `execa` with `CI=1`, 30 s default timeout, `reject:false` so tests can assert on non-zero exit codes explicitly.
- `FIXTURES_DIR` / `CLI_BIN` exported for any future helpers that need to introspect the layout.

### vitest globalSetup (`src/__tests__/helpers/global-setup.ts`)
- Walks `src/` (skipping `__tests__` + `__fixtures__`) and compares newest `mtimeMs` against `dist/index.js`.
- Runs `pnpm build` only when dist is missing or stale — keeps the unit suite fast while guaranteeing e2e tests see a fresh bin in CI.
- Registered via `vitest.config.ts` `globalSetup: [...]`. `testTimeout` + `hookTimeout` bumped to 30 s for the e2e spawn path.

### E2e test matrix — 4 files, 2 tests each
| File | Asserts |
|------|---------|
| `init-next-app.e2e.test.ts` | route handler at `app/api/uploadkit/[...uploadkit]/route.ts`; `UploadKitProvider` wraps `{children}` inside `<body>`; original `inter.className` preserved; `.env.local` merged with `UPLOADKIT_API_KEY=uk_test_placeholder`; `.uploadkit-backup/` exists + gitignored; second run prints "already configured" + zero mtime deltas |
| `init-sveltekit.e2e.test.ts` | `+server.ts` route exists with `RequestHandler`; `src/lib/uploadkit.ts` client stub references `@uploadkitdev/core`; `.env` has un-prefixed key; `+layout.svelte` NOT mutated; idempotent second run |
| `init-remix.e2e.test.ts` | `api.uploadkit.$.tsx` with exported `action` + `loader`; `root.tsx` wrapped with provider between `<body>` and `<Outlet>`; `.env` populated; idempotent second run |
| `init-vite-react.e2e.test.ts` | Provider wraps `<App/>` inside `<StrictMode>` in `src/main.tsx`; `VITE_UPLOADKIT_API_KEY` prefixed env; stderr includes Vite + backend/server/byos warning; idempotent second run |

All four use `describe.sequential(...)` to prevent tmpdir pressure from concurrent execution.

### Add + restore roundtrip (`e2e/add-and-restore.e2e.test.ts`)
Three tests:
1. **Active**: init → `restore --yes` returns a next-app fixture to a virgin state (route handler deleted, layout byte-identical to original, env + client stub deleted).
2. **Skipped**: full init → add dropzone → restore(add, by timestamp) → restore(init, by timestamp) roundtrip — skipped pending the add-template path defect (see Deferred Issues).
3. **Regression guard**: asserts `uploadkit add` currently fails with `Template not found for component alias`. If `add` ever starts succeeding against the compiled bin, this test throws a loud error telling the next maintainer to flip the `.skip` above.

## Deviations from Plan

### Auto-fixed / In-scope adjustments

**1. [Rule 3 — Blocking] Replaced `--latest --yes` with `--yes` + `--timestamp`**
- **Found during:** Task 2
- **Issue:** The plan references `restore --latest --yes`, but `restore` has no `--latest` flag. Its flag set is `--yes` (auto-picks newest non-interactively) and `--timestamp <iso>` (picks a specific session).
- **Fix:** The active roundtrip uses `--yes` (semantically "latest"). The seeded full roundtrip uses `--timestamp` to deterministically address the older session, because session directories are NOT removed after replay — chaining two `--yes` calls would replay the same manifest twice.
- **Files modified:** `packages/uploadkit-cli/src/__tests__/e2e/add-and-restore.e2e.test.ts`
- **Commit:** `e40dee8`

### Deferred Issues (out-of-scope per plan boundary)

**2. [Defect — NOT FIXED] `add` subcommand broken in the compiled bin**
- **Found during:** Task 2 first run
- **Location:** `packages/uploadkit-cli/src/add/insert-component.ts` lines 32–36
- **Bug:** Second entry in `TEMPLATE_CANDIDATES` resolves to `<pkg>/add-templates` (because `MODULE_DIR` is `<pkg>/dist` in the bundle) instead of `<pkg>/dist/add-templates` where `tsup`'s `onSuccess` hook copies the component templates. Result: every `uploadkit add <alias>` invocation on the published bin exits 1 with `Template not found for component alias`.
- **Why NOT fixed here:** The task explicitly forbids edits under `src/add/`. Logged to `deferred-items.md` with the recommended one-line change (swap second candidate to `resolve(MODULE_DIR, 'add-templates')`) plus a pointer to the seeded `it.skip` test that becomes the full-roundtrip proof as soon as it's addressed.
- **Regression guard committed** (`e40dee8`): asserts the current failure mode so an accidental fix cannot sneak through without reviewer attention.
