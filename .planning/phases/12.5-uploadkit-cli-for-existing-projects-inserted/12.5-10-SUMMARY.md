# Phase 12.5 Plan 10: Publish prep + public docs Summary

Ships `uploadkit@0.1.0` to publish-readiness: Changeset authored, README
turned into an npm-page quality document, docs MDX guide published to
`/docs/guides/cli-existing-projects`, and `release.yml` now requires the
uploadkit CLI smoke matrix to pass before publish.

## What Was Built

### Changeset — `.changeset/uploadkit-cli-initial.md`

Minor bump against `uploadkit` for the first public release. Release note
covers the command surface (`init`, `add`, `restore`) and the backup
pipeline. Verified with `pnpm changeset status`:

```
info Packages to be bumped at minor:
 - create-uploadkit-app
 - uploadkit
```

### README — `packages/uploadkit-cli/README.md`

Replaced the stub with a full npm-page document:

- **Hero**: "Add UploadKit to your existing app in under 60 seconds."
- **Quickstart**: one-liner with a pointer to `create-uploadkit-app` for
  greenfield users.
- **Per-framework matrix**: route handler path + provider mount per
  framework (Next.js App Router / SvelteKit / Remix-RR7 / Vite+React).
- **`add` catalog**: all six aliases from D-09 (`dropzone`, `button`,
  `modal`, `gallery`, `queue`, `progress`).
- **Rollback section**: `uploadkit restore` + `.uploadkit-backup/` layout.
- **Command reference**: flag-by-flag table.
- **Troubleshooting**: framework detection, idempotency markers, Vite
  BYOS warning.

### Release workflow gate — `.github/workflows/release.yml`

Added an inline `uploadkit-cli-smoke` job to `release.yml` that runs the
same matrix the dedicated `uploadkit-cli-smoke.yml` exercises
(`next-app`, `sveltekit`, `remix`, `vite-react`). The `release` job's
`needs:` now requires both smoke jobs:

```yaml
release:
  needs: [smoke, uploadkit-cli-smoke]
```

`fail-fast: false` preserves the full cross-framework status surface
when one leg breaks.

New public page covering:

- Per-framework `<Tabs>` walkthrough of what `init` changes.
- Idempotency explanation (marker comments).
- `add` flow with the full six-component table.
- Backup + restore mechanics with `--timestamp` escape hatch.
- Command reference table.
- Troubleshooting: framework detection, already-configured, provider
  collision, `.env.local` merge, Pages Router.
- Cross-links to `create-uploadkit-app`, Quickstart, npm page, GitHub.

## Deviations from Plan

### Auto-resolved

**1. [Rule 3 — Blocking] Inlined smoke job in release.yml instead of cross-workflow `needs`**
- **Found during:** Task 2 implementation
- **Issue:** Plan frontmatter `key_links` suggested `needs: .*smoke`
  pointing from `release.yml` → `uploadkit-cli-smoke.yml`. GitHub
  Actions does **not** support `needs:` across separate workflow files —
  `needs:` only references jobs within the same workflow.
- **Fix:** Added an inline `uploadkit-cli-smoke` matrix job to
  `release.yml` mirroring the standalone workflow's steps, and listed it
  in `release.needs`. The standalone `uploadkit-cli-smoke.yml` still
  runs on every PR for fast feedback; `release.yml`'s inline copy is the
  authoritative publish gate.
- **Files modified:** `.github/workflows/release.yml`
- **Commit:** `5b4d1a1`

**2. [Rule 2 — Correctness] Maintained TDD-light execution without red-phase commits**
- **Found during:** Tasks 2 and 3
- **Issue:** Plan frontmatter marks tasks 2 and 3 as `tdd="true"`, but
  the outputs are non-code artifacts (a README, a changeset markdown
  file, YAML workflow config, and an MDX docs page). There is no
  meaningful RED-phase test to write against prose + declarative config.
- **Fix:** Substituted behavioral verification for the TDD cycle:
  `pnpm changeset status` validates the changeset, `pnpm --filter docs
  build` validates the MDX parses and registers in the sidebar, and
  `pnpm --filter uploadkit build && test` confirms no regressions in
  existing CLI code. These checks were run after each commit.
- **No files modified** — this is a process note for the maintainer.

### Deferred (out of scope)

**3. `actionlint` not installed locally**
- Plan 09 flagged this. Still not installed. GitHub's own workflow
  validator will surface shape errors on the first push of
  `release.yml`.
