<objective>
Implement the complete upload pipeline: single-file presigned URL request/complete flow and multipart upload init/complete/abort endpoints. This is the core product — the presigned URL pipeline that lets developers upload files directly to R2.

Purpose: This is the primary value delivery of Phase 3. Without these endpoints, the SDK (Phase 4) has nothing to call.
Output: 5 route files covering the full upload lifecycle (request, complete, multipart init, multipart complete, multipart abort)

UPLD-05 (upload progress) server deliverable: The presigned PUT URL and presigned part URLs returned by these endpoints enable the client SDK to use XHR with `onprogress` callbacks. The server's job is done by returning the URLs — the actual progress tracking is implemented in the SDK (Phase 4).

UPLD-07 (automatic retry) server deliverable: The error format from Plan 03-01 (`errors.ts`) establishes the retryability contract: 5xx and 429 responses are retryable, 4xx responses are non-retryable. These endpoints must honour that contract by returning correct status codes. The SDK (Phase 4) implements the retry loop.
</objective>

Add `export const runtime = 'nodejs';` at top of file (per API-01).

Export `POST` handler wrapped with `withApiKey(handler, true)` (useUploadLimit=true for 30 req/min).

Handler steps (in exact order per research Pattern 3):
1. Parse body with `UploadRequestSchema.safeParse(await req.json())` — on failure return `serializeValidationError(parsed.error)`
2. Look up FileRouter: `FileRouter.findOne({ projectId: ctx.project._id, slug: parsed.data.routeSlug })` — if not found, return `serializeError(new NotFoundError('FileRouter', parsed.data.routeSlug))`
3. Validate file size against BOTH fileRouter.maxFileSize AND `TIER_LIMITS[ctx.tier].maxFileSizeBytes` — use `Math.min()` as the effective limit. On violation: `serializeError(new TierLimitError('file size'))`
4. Validate contentType against fileRouter.allowedTypes (skip if allowedTypes is empty array — means all types allowed). On violation: return 400 with code `INVALID_FILE_TYPE`
5. Check storage quota: look up `UsageRecord.findOne({ userId: ctx.project.userId, period: new Date().toISOString().slice(0, 7) })` — if `(record?.storageUsed ?? 0) + parsed.data.fileSize > TIER_LIMITS[ctx.tier].maxStorageBytes`, return `serializeError(new TierLimitError('storage'))`
6. Check upload count quota: if `(record?.uploads ?? 0) >= TIER_LIMITS[ctx.tier].maxUploadsPerMonth`, return `serializeError(new TierLimitError('monthly uploads'))`
7. Generate R2 key: `import { nanoid } from 'nanoid'; const key = \`${ctx.project._id}/${parsed.data.routeSlug}/${nanoid()}/${parsed.data.fileName}\``
8. Generate presigned URL: `const uploadUrl = await generatePresignedPutUrl({ key, contentType: parsed.data.contentType, contentLength: parsed.data.fileSize, expiresIn: 900 })`
9. Insert File record: `const file = await File.create({ key, name: parsed.data.fileName, size: parsed.data.fileSize, type: parsed.data.contentType, url: \`${CDN_URL}/${key}\`, status: 'UPLOADING', metadata: parsed.data.metadata, projectId: ctx.project._id })`
10. Return `NextResponse.json({ fileId: file._id.toString(), uploadUrl, key, cdnUrl: file.url })` with status 200

Wrap entire handler body in try/catch, returning `serializeError(err)` on catch.

Handler steps:

2. Perform the SAME validation as upload/request: FileRouter lookup, file size check (use `Math.min(fileRouter.maxFileSize, TIER_LIMITS[ctx.tier].maxFileSizeBytes)`), content type check, storage quota check, upload count check

4. Generate R2 key: same pattern as single upload
5. Calculate part count: `const PART_SIZE = 5 * 1024 * 1024; const partCount = Math.ceil(parsed.data.fileSize / PART_SIZE);`
6. Call `CreateMultipartUploadCommand`: `const { UploadId } = await r2Client.send(new CreateMultipartUploadCommand({ Bucket: R2_BUCKET, Key: key, ContentType: parsed.data.contentType }))`
7. Generate presigned URLs for ALL parts up-front: loop from `partNumber = 1` to `partCount`, for each call `getSignedUrl(r2Client, new UploadPartCommand({ Bucket: R2_BUCKET, Key: key, UploadId, PartNumber: partNumber }), { expiresIn: 3600 })` — use 1 hour expiry for multipart parts since large files take longer
8. Insert File record with `status: 'UPLOADING'` and `uploadId: UploadId`
9. Return `NextResponse.json({ fileId: file._id.toString(), uploadId: UploadId, key, parts: parts.map((url, i) => ({ partNumber: i + 1, uploadUrl: url })) })`

Import `CreateMultipartUploadCommand`, `UploadPartCommand` from `@aws-sdk/client-s3` and `getSignedUrl` from `@aws-sdk/s3-request-presigner`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> upload/request | Untrusted: file metadata, size claims, content type |
| Client -> upload/complete | Untrusted: fileId could reference another project's file |
| Client -> multipart/complete | Untrusted: ETag array comes from client (collected from R2 responses) |
| API -> R2 | Trusted: presigned URL generation + HEAD verification server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Spoofing | upload/request | mitigate | Content-Type and Content-Length locked in presigned URL signature (Phase 1 design); client cannot upload different type |
| T-03-08 | Tampering | upload/complete | mitigate | HEAD verification in R2 confirms file actually exists before marking UPLOADED |
| T-03-09 | Elevation | upload/complete | mitigate | File lookup scoped by projectId from withApiKey context — cannot confirm another project's file |
| T-03-10 | Denial of Service | upload/request | mitigate | Tier limit checks BEFORE presigning — storage quota, upload count, file size all validated |
| T-03-11 | Info Disclosure | multipart/init | mitigate | Presigned part URLs expire in 1 hour; cannot be used to list or delete objects |
| T-03-12 | Tampering | multipart/complete | accept | ETags from client cannot be forged — R2 validates them during CompleteMultipartUpload; invalid ETags cause R2 error |
</threat_model>
