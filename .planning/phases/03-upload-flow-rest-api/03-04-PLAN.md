**1. Create `apps/api/src/app/api/v1/logs/route.ts` — GET (API-07):**

Add `export const runtime = 'nodejs';`

Export `GET` wrapped with `withApiKey(handler)`.

Handler steps:
1. Parse query params with `LogsQuerySchema`: `const url = new URL(req.url); const parsed = LogsQuerySchema.safeParse({ since: url.searchParams.get('since'), limit: url.searchParams.get('limit') });`
2. On parse failure, return `serializeValidationError(parsed.error)`
3. Query: `File.find({ projectId: ctx.project._id, createdAt: { $gt: parsed.data.since } }).sort({ createdAt: -1 }).limit(parsed.data.limit).lean()`
4. Return `NextResponse.json({ logs: files.map(f => ({ id: f._id, key: f.key, name: f.name, size: f.size, type: f.type, status: f.status, createdAt: f.createdAt, updatedAt: f.updatedAt })), count: files.length })`

This endpoint is designed for 5-second polling from the dashboard — the `since` param is the timestamp of the last received event.

**2. Create `apps/api/src/app/api/cron/cleanup/route.ts` — GET (UPLD-09, D-06):**

Add `export const runtime = 'nodejs';`
Add `export const dynamic = 'force-dynamic';` (prevent static caching of cron response).

This is NOT wrapped with `withApiKey` — it uses CRON_SECRET header auth instead.

Handler steps:
1. Validate `x-cron-secret` header: `if (req.headers.get('x-cron-secret') !== process.env.CRON_SECRET) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });` — also accept Vercel's `authorization: Bearer <CRON_SECRET>` header for Vercel Cron compatibility
2. `await connectDB()`
3. Find stale files: `const cutoff = new Date(Date.now() - 60 * 60 * 1000); const staleFiles = await File.find({ status: 'UPLOADING', createdAt: { $lt: cutoff } });`
4. Process each file with `Promise.allSettled`:
   - If `file.uploadId` is set (multipart in progress): call `AbortMultipartUploadCommand({ Bucket: R2_BUCKET, Key: file.key, UploadId: file.uploadId })` — catch and log errors (upload may already be aborted)

   - Then `File.deleteOne({ _id: file._id })`
5. Log results: `console.log(\`[cleanup] Processed ${staleFiles.length} stale uploads\`)`
6. Return `NextResponse.json({ cleaned: staleFiles.length, timestamp: new Date().toISOString() })`

Import `DeleteObjectCommand`, `AbortMultipartUploadCommand` from `@aws-sdk/client-s3`.
Import `connectDB`, `File` from `@uploadkit/db`.

**Create `apps/api/src/app/api/v1/webhooks/qstash-dlq/route.ts` — POST (D-10):**

This route is called by QStash when all retries are exhausted for a webhook delivery. It records the failure on the File record so the dashboard can surface it.

Add `export const runtime = 'nodejs';`

This endpoint is NOT protected by `withApiKey` — it is called by QStash, not by a developer API key. Authenticate using the `Upstash-Signature` header verification from `@upstash/qstash`.

Handler steps:
1. Verify the QStash signature to confirm the call is genuinely from QStash (not a spoofed request):
   ```typescript
   import { Receiver } from '@upstash/qstash';
   const receiver = new Receiver({
     currentSigningKey: process.env.QSTASH_CURRENT_SIGNING_KEY ?? '',
     nextSigningKey: process.env.QSTASH_NEXT_SIGNING_KEY ?? '',
   });
   const signature = req.headers.get('upstash-signature') ?? '';
   const body = await req.text();
   const isValid = await receiver.verify({ signature, body }).catch(() => false);
   if (!isValid) {
     return NextResponse.json({ error: 'Invalid signature' }, { status: 401 });
   }
   ```
   If `QSTASH_CURRENT_SIGNING_KEY` is not set (dev environment), skip verification and log a warning.

2. Parse the body (the original webhook payload QStash was trying to deliver):
   ```typescript
   const payload = JSON.parse(body) as { file?: { id?: string }; projectId?: string };
   const fileId = payload?.file?.id;
   ```

3. If `fileId` is present, update the File record to record the failure:
   ```typescript
   await connectDB();
   await File.findByIdAndUpdate(fileId, {
     $set: { webhookFailedAt: new Date() },
   });
   ```
   This requires adding `webhookFailedAt?: Date` to the `IFile` interface and schema in `packages/db/src/models/file.ts` if not already present.

4. Log the failure: `console.error('[qstash-dlq] Webhook delivery exhausted for file:', fileId)`

5. Return `NextResponse.json({ ok: true })` — QStash expects a 2xx to acknowledge the DLQ callback.

Add `QSTASH_CURRENT_SIGNING_KEY` and `QSTASH_NEXT_SIGNING_KEY` to `.env.example` with a comment: `# QStash signing keys — from Upstash Console > QStash > Signing Keys`.

Note: The `webhookFailedAt` field must be added to `packages/db/src/models/file.ts` in this task. The `api_keys` collection rename in Task 1 of Plan 03-01 established the pattern — add it the same way (optional Date field, no default).

| Boundary | Description |
|----------|-------------|
| Cron caller -> cleanup endpoint | Must verify CRON_SECRET to prevent unauthorized cleanup |
| Client -> logs endpoint | Untrusted: `since` param could be manipulated |
| QStash -> qstash-dlq endpoint | Must verify Upstash-Signature to prevent spoofed DLQ callbacks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-18 | Spoofing | cron/cleanup | mitigate | CRON_SECRET header validation — rejects requests without valid secret |
| T-03-19 | Denial of Service | cron/cleanup | mitigate | Cron runs hourly (not on demand); processes stale files in parallel with allSettled (no cascading failure) |
| T-03-20 | Elevation | logs endpoint | mitigate | Logs scoped by projectId from withApiKey context — cannot view other project's logs |
| T-03-21 | Tampering | cron/cleanup | accept | Cleanup only deletes UPLOADING files older than 1 hour — no risk to completed uploads |
| T-03-22 | Spoofing | webhooks/qstash-dlq | mitigate | Upstash-Signature HMAC verification via @upstash/qstash Receiver — rejects calls not from QStash |
| T-03-23 | Tampering | webhooks/qstash-dlq | mitigate | DLQ only sets webhookFailedAt timestamp — no destructive action; payload fileId validated before DB update |
