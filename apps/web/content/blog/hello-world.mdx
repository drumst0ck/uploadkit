Welcome to the **UploadKit** blog. This is where we'll publish long-form writing that doesn't fit in the [changelog](/changelog) — technical deep dives, product decisions, and tutorials for the developers who are shipping with UploadKit every day.

## What you can expect

We're writing for developers, so we'll keep the marketing fluff to zero. Every post here should save you time or teach you something.

- **Tutorials** — end-to-end guides for specific problems ("upload a 20 GB video from a mobile browser", "swap your S3 for R2 in production without downtime").
- **Comparisons** — honest head-to-heads with UploadThing, Uploadcare, Cloudinary, and others. Including where _they_ are better.
- **Engineering** — notes from building the platform. What broke, what we learned, what we changed our minds about.
- **Changelog** highlights — deeper context on shipped features that deserve more than a bullet point.

## Built with MDX, styled like the rest of the app

Every post is a plain `.mdx` file in the repo. Frontmatter drives routing, SEO, and the category filter. Code blocks use [Shiki](https://shiki.style/) for syntax highlighting — here's a fully-typed UploadKit route handler:

```ts
import { createRouteHandler } from '@uploadkitdev/next'
import { fileRouter } from '@/server/uploadkit'

export const { GET, POST } = createRouteHandler({
  router: fileRouter,
  config: {
    token: process.env.UPLOADKIT_TOKEN!,
    // Optional: bring your own S3/R2 bucket
    mode: 'byos',
  },
})
```

You can inline things like `UploadButton` or `presignPutObject` and they'll render in a themed code pill — just like the component reference in the docs.

> Every post ships with a dynamic Open Graph image, JSON-LD `Article` schema, and an entry in the sitemap and RSS feed. Nothing else to configure.
