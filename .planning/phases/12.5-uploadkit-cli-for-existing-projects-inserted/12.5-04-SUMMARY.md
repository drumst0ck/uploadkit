# Phase 12.5 Plan 04: `init` for Next.js App Router Summary

Shipped the Next.js App Router flavour of `npx uploadkit init` — the reference implementation that plan 12.5-05 will clone for SvelteKit / Remix / Vite. `init` now detects the host framework, opens a backup session, wraps `<body>`'s children with `<UploadKitProvider>`, creates the canonical route handler + client stub, merges `UPLOADKIT_API_KEY` into `.env.local`, and installs the SDK packages via the host's package manager. Second invocation short-circuits with "already configured" and zero filesystem changes. End-to-end smoke run (cloned fixture → `node dist/index.js init --yes --skip-install`) completes in under 150ms.

## What was built

### Task 1 — `installPackages` helper + init orchestrator skeleton

**`src/init/install-deps.ts`** — wraps `execa` with a per-package-manager argv table:

| PM    | Command        |
| ----- | -------------- |
| pnpm  | `pnpm add …`   |
| npm   | `npm install …`|
| yarn  | `yarn add …`   |
| bun   | `bun add …`    |

`skipInstall: true` is a pure no-op (intended for CI and unit tests). Empty `pkgs` is also a no-op — keeps callers from having to guard. Errors thrown by execa on non-zero exit propagate unchanged; the orchestrator catches them and finalises the backup so nothing is orphaned.

**`src/commands/init.ts`** — orchestrator. Flow:

1. `detectFramework(cwd)` → verdict + detection metadata.
2. On `unknown` / `next-pages` → stderr + exit 1 without touching any file.
3. Look up the per-framework impl in `IMPLS`; if absent, print "coming soon" (plan 12.5-05's responsibility) and exit 1.
4. Construct `InitContext` from parsed flags + detection.
5. `createBackupSession(root)` — deferred-write session.
6. Run the impl. On success: `session.finalize()` + human-readable summary. On error: best-effort finalize so partial backups survive, then surface the error.

`src/init/types.ts` defines the small shared surface consumed by each per-framework impl: `InitContext`, `InitResult`, `InitImpl`. Keeping these in a dedicated file means plan 12.5-05 can implement SvelteKit/Remix/Vite without coupling to Next.

### Task 2 — Next.js App Router implementation

**`src/init/next-app.ts`** — the real work. Order:

1. **Idempotency gate** — if `app/layout.tsx` has marker comments AND the `@uploadkitdev/react` import is present AND the route handler file exists, return `{skipped: true}` with empty lists. Orchestrator prints the yellow "already configured" line and does NOT finalize the session (no empty manifest on disk).
2. **Backup** — `session.save(app/layout.tsx)` copies the original into `.uploadkit-backup/<timestamp>/app/layout.tsx` BEFORE any mutation.
3. **Rewrite layout** — `addImport(source, { from: '@uploadkitdev/react', specifiers: ['UploadKitProvider'] })` then `wrapChildrenOf(src, { parentTag: 'body', wrapperOpen: '<UploadKitProvider>', wrapperClose: '</UploadKitProvider>' })`. `wrapChildrenOf` defaults to `markerStyle: 'jsx'` so markers land as `{/* uploadkit:start … */}` — valid JSX comments that render nothing in the DOM.
4. **Create route handler** — `app/api/uploadkit/[...uploadkit]/route.ts` from `templates/next-route-handler.ts.tpl`. `writeIfAbsent` short-circuits if the user already has a file there (defensive: covers manual half-installs).
5. **Create client stub** — `lib/uploadkit.ts` from `templates/next-uploadkit-client.ts.tpl`. Same `writeIfAbsent` guard.
6. **Merge .env.local** — `UPLOADKIT_API_KEY=uk_test_placeholder` appended only if absent (mergeEnv preserves existing values + comments, per 12.5-03).
7. **Install packages** — `installPackages(detection.packageManager, root, ['@uploadkitdev/next@latest', '@uploadkitdev/react@latest'])`. Skipped when `--skip-install`; in that case the CLI prints a dim hint with the exact PM command so the user can run it manually.

The fixture at `__fixtures__/projects/next-app/app/layout.tsx` was upgraded from a two-line stub to a realistic Next.js 16 layout (font import, Metadata export, body with className, html lang) so tests exercise the codemods against something representative.

### Exact SDK helper name referenced

Verified against `packages/next/src/index.ts`:

- **`createUploadKitHandler`** — NOT `createRouteHandler` as hinted in the plan's `<interfaces>` block. Signature:
  ```ts
  createUploadKitHandler<TRouter extends FileRouter>(
    config: UploadKitHandlerConfig<TRouter>
  ```

- **`FileRouter`** type — from `@uploadkitdev/next`. Used in the client stub template for the `satisfies FileRouter` annotation.

Documented as Deviation 1 below.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] JSX-inserted line-comment markers render as DOM text**

- **Found during:** Task 2 smoke test. The initial implementation inserted `// uploadkit:start` / `// uploadkit:end` as literal lines between `<body>` and `<UploadKitProvider>`. Syntactically valid (TS accepts JSX text content) — but in the browser those strings render as visible page text, corrupting the UI.
- **Fix:** Added `JSX_MARKER_START = '{/* uploadkit:start — do not edit this block manually */}'` + `JSX_MARKER_END = '{/* uploadkit:end */}'` to `codemods/markers.ts`. Upgraded `wrapChildrenOf` with a `markerStyle: 'jsx' | 'line'` option defaulting to `'jsx'` (the safe choice inside JSX children — which is the only position `wrapChildrenOf` ever targets today). Line-comment markers remain available for non-JSX insertion sites (route handler file, client stub — both plain `.ts`). `hasMarkers` was widened to match the semantic `uploadkit:start` / `uploadkit:end` key so idempotency works regardless of comment form.
- **Files modified:**
  - `packages/uploadkit-cli/src/codemods/markers.ts`
  - `packages/uploadkit-cli/src/codemods/jsx-wrap.ts`
  - `packages/uploadkit-cli/src/codemods/index.ts`
  - `packages/uploadkit-cli/src/__tests__/backup.test.ts` (updated assertion to use the semantic key instead of `MARKER_START`)
- **Commit:** `c3c3f60 feat(12.5-04): initNextApp implementation + JSX-safe markers`
- **Scope note:** This modifies a codemod from 12.5-03. The plan's `<action>` block for Task 2 explicitly permits adjustments (magicast fallback behavior + SDK name verification), but marker-form wasn't called out. Fix is in scope because the bug only manifests when a caller uses `wrapChildrenOf` for a JSX-children position — which is exactly what 12.5-04 introduces. The 12.5-03 test that asserted on `MARKER_START` was updated to assert on the semantic `uploadkit:start` key so future marker-form changes don't need to touch it.

**2. [Rule 2 — Missing critical functionality] Template files not shipped in dist**

- **Found during:** Build output review. `tsup` compiles `src/index.ts` but doesn't copy `.tpl` files. Without the copy, the published npm `bin` would fail at runtime with ENOENT when loading templates.
- **Fix:** Added `onSuccess` hook to `tsup.config.ts` that `cpSync`s `src/init/templates/` into `dist/templates/`. The runtime resolver (`resolve(fileURLToPath(import.meta.url), '..', 'templates')`) correctly points at `src/init/templates` during source execution (vitest) and `dist/templates` after the bundle.
- **Files modified:** `packages/uploadkit-cli/tsup.config.ts`
- **Commit:** `c3c3f60 …`

### Inside Claude's discretion

**3. `lib/uploadkit.ts` stub is create-only.** The plan says "create `lib/uploadkit.ts` stub if absent" — I implemented this literally. A user who already has a client file won't have it clobbered. Adds a test (`preserves existing lib/uploadkit.ts`) asserting the guard.

**4. `.env.local` file creation is tracked in the backup manifest.** When `.env.local` doesn't exist, `mergeEnv` creates it AND `next-app.ts` records a `create` entry in the backup session — so `restore` can delete the CLI-created env file instead of leaving orphaned placeholder values.
