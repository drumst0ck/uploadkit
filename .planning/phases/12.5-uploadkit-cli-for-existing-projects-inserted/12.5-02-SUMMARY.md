# Phase 12.5 Plan 02: Framework Detector + Package-Manager Detector Summary

Shipped the detection layer every downstream plan in this phase consumes. `detectFramework(cwd)` walks up from any directory to the nearest `package.json`, applies a first-match-wins rule engine, and returns a typed `DetectionResult` with the framework verdict, absolute project root, inferred package manager, and a diagnostic `reason` string on refusal paths. `detectPackageManager(cwd)` is a pure lockfile-first function that falls back to the npm user-agent env var and finally to `npm`. Both are exported from `packages/uploadkit-cli/src/detect/` and exhaustively tested against 6 fixture projects plus a suite of edge cases.

## What was built

### Task 1 — Framework detector, types, and 6 fixture projects

`detect/types.ts` defines the shared surface — `Framework`, `PackageManager`, `DetectionResult` — consumed by plans 04, 05, 06, 07. Keeping types in their own zero-runtime module means plans that only need the type (e.g. refusal handling in `init`) can import it without pulling the detection logic into their graph.

`detect/framework.ts` implements `detectFramework(cwd)`:

1. Walk up from `cwd` via `dirname`-loop, stopping when a `package.json` file is found or when the filesystem root is reached.
2. Parse `dependencies`, `devDependencies`, `peerDependencies`, `optionalDependencies` into a merged bag.
3. Apply rules in D-04 order:
   - `next` + (`app/` or `src/app/`) present → `next-app`
   - `next` + `pages/_app.*` present (no `app/`) → `next-pages` with reason "Pages Router not supported — migrate to App Router"
   - `@sveltejs/kit` → `sveltekit`
   - any `@remix-run/*` dep OR `@react-router/dev` → `remix`
   - `vite` AND `react` → `vite-react`
   - otherwise → `unknown` with a reason enumerating what was missing
4. Return a `DetectionResult` plus the result of `detectPackageManager(root)`.

The 6 fixtures under `src/__fixtures__/projects/` are the happy-path matrix:

| Fixture | Dependencies | Marker file(s) | Lockfile | Expected verdict | Expected PM |
| --- | --- | --- | --- | --- | --- |
| `next-app` | `next`, `react`, `react-dom` | `app/layout.tsx` | `pnpm-lock.yaml` | `next-app` | `pnpm` |
| `next-pages` | `next`, `react`, `react-dom` | `pages/_app.tsx` | `package-lock.json` | `next-pages` (refused) | `npm` |
| `sveltekit` | `@sveltejs/kit`, `svelte`, `vite` | `svelte.config.js` | `yarn.lock` | `sveltekit` | `yarn` |
| `remix` | `@remix-run/node`, `@remix-run/react`, `@remix-run/dev` | — | `bun.lock` | `remix` | `bun` |
| `vite-react` | `react`, `vite`, `@vitejs/plugin-react` | `vite.config.ts` | `pnpm-lock.yaml` | `vite-react` | `pnpm` |
| `unknown` | `express` only | — | — | `unknown` (with reason) | `npm` (or user-agent) |

Each fixture intentionally varies the package manager so a single test call asserts both framework and PM in one shot. The `unknown` fixture deliberately has no lockfile to exercise the user-agent / `npm` fallback.

### Task 2 — Package-manager detector + integration test suite

`detect/package-manager.ts` ports the idea from `packages/create-uploadkit-app/src/pm.ts` but swaps the default order. Phase 12's greenfield CLI defaults to `pnpm` because the monorepo uses pnpm; this CLI runs INSIDE user projects where assuming pnpm would be wrong — so when no signal is found, it defaults to `npm` (the lowest-common-denominator that is safe everywhere).

Priority:
1. `pnpm-lock.yaml` → `pnpm`
2. `bun.lock` or `bun.lockb` (scanned via `readdirSync` in case only the binary variant exists) → `bun`
3. `yarn.lock` → `yarn`
4. `package-lock.json` → `npm`
5. `process.env.npm_config_user_agent` first token's name part (`pnpm/9.0.0 …` → `pnpm`)
6. Fallback → `npm`

`__tests__/detect.test.ts` runs 33 tests across three describe blocks:
- **fixture matrix** (6 tests) — one assertion per fixture for both framework and PM.
- **extra rules** (6 tests via tmpdirs) — walk-up from nested path; `@react-router/dev` as Remix; next with neither `app/` nor `pages/_app.*` → unknown diagnostic; empty-deps → unknown listing missing deps; vite-without-react; `src/app` also counts for next-app.
- **detectPackageManager** (7 tests via tmpdirs) — lockfile priority (pnpm wins over yarn+npm), bun via `bun.lock` and `bun.lockb`, yarn, npm, user-agent fallback, npm ultimate fallback.

## Deviations from Plan

None of substance. Two choices inside Claude's discretion:

1. **PM default fallback is `npm`, not `pnpm`.** The plan specifies "Final fallback: 'npm'". Phase 12's `pm.ts` defaults to `pnpm` for its own (different) use case — I kept this CLI's default at `npm` because `uploadkit init` runs inside unknown user projects where assuming pnpm would be hostile. Explicitly documented in the module's JSDoc.
2. **`optionalDependencies` and `peerDependencies` are merged alongside deps/devDeps.** The plan says "parse deps + devDeps"; widening this avoids false negatives when a user puts `next` in peer deps (rare, but some monorepo templates do this). Doesn't affect any fixture verdict.

Both tasks' `<done>` criteria are met.

## Edge cases worth remembering

- **Monorepo walk-up**: `detectFramework` walks up until it finds a `package.json`. If invoked from a nested directory that has no `package.json` in its chain, it falls back to the resolved cwd. This is the correct behavior for the common case (user runs `uploadkit init` from a subfolder by mistake), and downstream plans should still show the detected `root` to the user before mutating files (risk 2 in the phase plan).
- **Mixed `app/` + `pages/_app.*`**: `next-app` wins. Some projects still have a legacy `pages/` dir for incremental migration; we prefer the supported path.
- **`next` in deps but no router files**: returns `unknown` with a Next-specific reason asking the user to run from the project root. This is safer than silently picking `next-app` when no layout exists.
- **bun lockfile variants**: `bun.lock` (text) and `bun.lockb` (binary) both trip bun detection; we `readdirSync` once rather than stat'ing each candidate.
