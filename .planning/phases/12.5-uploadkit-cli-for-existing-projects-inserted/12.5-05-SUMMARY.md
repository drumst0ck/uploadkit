# Phase 12.5 Plan 05: `init` for SvelteKit, Remix/RR7, Vite+React Summary

Extended `npx uploadkit init` to the remaining three supported frameworks, completing the dispatch table in `commands/init.ts`. Each new framework follows the same orchestrator contract (`InitImpl = (ctx, session) => Promise<InitResult>`) as `initNextApp` from plan 04, with framework-specific deltas documented below. End-to-end smoke against cloned fixtures confirms fresh-run correctness and second-run idempotency for all three.

## What was built

### Task 1 — SvelteKit + Remix init modules

**`src/init/sveltekit.ts`** — no `.svelte` mutation, only creates.

1. Idempotency: `+server.ts` AND `src/lib/uploadkit.ts` both exist → `{skipped: true}` with no backup session dir created.
2. Creates `src/routes/api/uploadkit/[...uploadkit]/+server.ts` from `sveltekit-server.ts.tpl` (imports `$lib/uploadkit` + `$env/dynamic/private`, exports GET/POST typed as `RequestHandler`).
3. Creates `src/lib/uploadkit.ts` from `sveltekit-client.ts.tpl` (calls `createUploadKit({ apiKey })` from `@uploadkitdev/core`).
4. Merges `.env` with `UPLOADKIT_API_KEY=uk_test_placeholder` (additive-only).
5. Installs `@uploadkitdev/core@latest` — no React package.

The fixture already had `svelte.config.js` + `package.json` with `@sveltejs/kit`; I added a minimal `src/routes/+layout.svelte` so the test can assert "we didn't touch it" (see truth #5 from the plan frontmatter).

**`src/init/remix.ts`** — mirrors `initNextApp` structurally but targets `app/root.tsx`.

1. Idempotency: markers in `app/root.tsx` AND `app/routes/api.uploadkit.$.tsx` exists.
2. Backs up `app/root.tsx` before mutation.
3. Rewrites root.tsx: `addImport('@uploadkitdev/react', ['UploadKitProvider'])` + `wrapChildrenOf({ parentTag: 'body', wrapperOpen: '<UploadKitProvider endpoint="/api/uploadkit">', … })`. `markerStyle` defaults to `'jsx'` — the markers land as valid JSX comments inside the `<body>` children.
4. Creates `app/routes/api.uploadkit.$.tsx` from `remix-route.tsx.tpl` (exports `action` + `loader`). The template is intentionally a scaffold with a `TODO:` pointing at the docs — the UploadKit Remix SDK helper is not yet published (deviation 2).
5. Merges `.env` with `UPLOADKIT_API_KEY=uk_test_placeholder`.
6. Installs `@uploadkitdev/react@latest` + `@uploadkitdev/core@latest`.

The fixture `package.json` uses `@remix-run/*` classic (not `react-router`) — both are detected as `remix` by the detector (12.5-02), and `wrapChildrenOf('body', ...)` targets `<body>` in either shape.

### Task 2 — Vite+React init + dispatcher wiring

**`src/init/vite-react.ts`** — client-only. Key differences from Remix:

1. No route handler creation (no server). The warning text explicitly tells the user a backend (or BYOS) is required.
2. Parent tag for `wrapChildrenOf` is `StrictMode`, not `body`. The fixture's `src/main.tsx` wraps `<App />` in `<StrictMode>…</StrictMode>` so the wrapper slots in between `<StrictMode>` and `<App />`.
3. If `<StrictMode>` is absent, we throw with a remediation message rather than heuristically rewriting `createRoot(...).render(…)` (which would be fragile across codebases).
4. Env key is `VITE_UPLOADKIT_API_KEY=uk_test_placeholder` — Vite requires the `VITE_` prefix to expose vars to the browser.
5. Emits the server-required warning on stderr AFTER install, so it's the last thing the user sees.
6. Installs only `@uploadkitdev/react@latest` (no `core` — the browser-safe surface comes through the provider's proxy client).

**`src/commands/init.ts`** — dispatcher. The `IMPLS` table now covers all 4 supported frameworks:

```ts
const IMPLS: Partial<Record<Framework, InitImpl>> = {
  'next-app': initNextApp,
  sveltekit: initSvelteKit,
  remix: initRemix,
  'vite-react': initViteReact,
};
```

The "coming soon" branch is now unreachable for supported verdicts — only `next-pages` and `unknown` hit the refusal paths (both return exit 1 before allocating a backup session).

## SDK helper name deviations encountered (as called out in `<output>`)

1. **SvelteKit has no first-party UploadKit SDK at the time of writing.** The generated `src/lib/uploadkit.ts` calls `createUploadKit` from `@uploadkitdev/core` (verified in `packages/core/src/index.ts` line 5). The `+server.ts` template calls `uploadkit.handle({ request, subpath })` — this method doesn't exist on the core client yet; the generated comment block points users at https://uploadkit.dev/docs. A future plan should either ship a `@uploadkitdev/sveltekit` helper or evolve `@uploadkitdev/core` to include a `handle()` method. Documented as Deviation 2.
2. **Remix has no first-party SDK either.** The generated `app/routes/api.uploadkit.$.tsx` is a scaffold with env-var validation and a `TODO:` forwarding note. This mirrors the `create-uploadkit-app/templates/remix/` pattern where the user composes an S3 presigner directly. Documented as Deviation 2.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing critical functionality] Vite main.tsx rewrite needed a safer anchor than raw regex over `createRoot(...).render(…)`**

- **Found during:** Task 2 implementation. The plan's `<action>` block suggested either extending `jsx-wrap.ts` with a `wrapRenderArgument` helper or using magicast inline. Both paths were fragile: real-world `src/main.tsx` files wrap the render argument in `<StrictMode>`, `<ThemeProvider>`, or `<BrowserRouter>` — no single heuristic finds the "right" anchor.
- **Fix:** Stayed with the existing `wrapChildrenOf` helper but pass `parentTag: 'StrictMode'`. If `<StrictMode>` is absent, throw a clear error telling the user to add it and re-run. This keeps the codemod surface tiny (no new API) and fails loudly rather than silently corrupting the file.
- **Files modified:** `packages/uploadkit-cli/src/init/vite-react.ts`
- **Commit:** `13447c5 feat(12.5-05): initViteReact + dispatcher wired for all 4 frameworks`

**2. [Inside discretion] SDK server-helper methods are stubs pointing at the docs**

- **Found during:** Template authoring. The plan implies a first-party helper exists per framework (e.g. `uploadkit.handle(...)` for Svelte, a Remix action helper). Only Next.js ships with `createUploadKitHandler` today (verified in `packages/next/src/index.ts`).
- **Fix:** For SvelteKit and Remix, generate a typed skeleton with env-var guards that compiles and boots, with a `TODO:` comment pointing at the docs. This matches how `create-uploadkit-app/templates/{sveltekit,remix}/` scaffold today (S3 presigner written inline, no abstraction). When the SDK adds framework-specific helpers, a follow-up plan can update the templates without changing the init flow.
- **Scope note:** Not a Rule 4 architectural change — we're not introducing a new dependency or API surface, just documenting a known gap in template content.

### Inside Claude's discretion

**3. Fixtures were expanded with realistic files.** The pre-existing fixtures only had `package.json` + lockfile + one config file — enough for detection but not for init. Added: `src/routes/+layout.svelte` (SvelteKit), `app/root.tsx` + `app/routes/_index.tsx` (Remix), `src/main.tsx` + `src/App.tsx` + `index.html` (Vite). This mirrors the approach plan 04 took for next-app/layout.tsx.

**4. Tests assert on the `@uploadkitdev/react` substring** (not the full `from '…'` form). Magicast's `generateCode` emits double quotes regardless of source file style, so `from "@uploadkitdev/react"` is what ends up in the file. Asserting on the substring decouples tests from magicast's quote style.

**5. `handle({ request, subpath })` in the SvelteKit server template.** This method shape is a placeholder that matches what a proper `@uploadkitdev/core` SDK method *should* look like when it ships. The template compiles because `@uploadkitdev/core` is not actually resolved in tests (we only read files). At runtime in a real project, the user will see a type error pointing them at the docs — which is the intended UX until the SDK method exists.
