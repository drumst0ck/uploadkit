    - Sections:
      1. What are file routes? (analogy: like API routes for file uploads — define accepted types, size limits, and server-side logic per upload category)

    **presigned-urls.mdx** — "How Uploads Work"
    - Title: "How Uploads Work"
    - Description: "UploadKit uses presigned URLs for direct-to-storage uploads — your server never proxies file bytes."
    - Sections:
      1. The presigned URL flow (numbered diagram-like explanation: Client requests URL -> Server validates + generates presigned URL -> Client uploads directly to R2/S3 -> Client confirms completion -> Server verifies + records)
      2. Why presigned URLs? (performance: no server bottleneck; cost: no egress through your server; scalability: CDN-backed storage)
      3. Multipart uploads (files >10MB automatically split into 5MB chunks, progress events per chunk, transparent to the developer)
      4. Retry and abort (exponential backoff with configurable maxRetries, AbortController support)
      5. Security (ContentType and ContentLength locked in signature to prevent type spoofing, per Phase 1 decision T-01-08)

    **byos.mdx** — "Bring Your Own Storage"
    - Title: "Bring Your Own Storage (BYOS)"
    - Description: "Use your own S3 or R2 bucket with zero frontend changes. No vendor lock-in."
    - Sections:
      1. What is BYOS? (use UploadKit SDK + components but store files in YOUR bucket)
      2. Setup (show S3CompatibleStorage config in createUploadKitHandler, env vars for credentials)
      3. Supported providers (Cloudflare R2, AWS S3, DigitalOcean Spaces, MinIO — any S3-compatible)
      4. Zero frontend changes (same UploadButton/UploadDropzone, no API change needed)
      5. Security model (credentials NEVER touch the browser — presigned URLs generated server-side only)
      6. When to use BYOS vs managed (comparison table: managed = zero config, BYOS = full control)
    - Use Tabs for R2/S3/MinIO config examples per D-02.

    **security.mdx** — "Security"
    - Title: "Security"
    - Description: "How UploadKit protects your uploads with API keys, validation, and rate limiting."
    - Sections:
      1. API key authentication (uk_live_ / uk_test_ prefixes, keys hashed with SHA256 and never stored in plaintext)
      2. File validation (server-side type checking, size limits enforced at presign time AND on upload completion)
      3. Presigned URL security (ContentType + ContentLength locked in signature, URL expires in 1 hour)
      4. Rate limiting (sliding window per API key via Upstash Redis)
      5. Tier limits (storage, bandwidth, upload count enforced per subscription tier)
      6. BYOS credential safety (server-side only, never in client bundle)
      7. Best practices (callout: never expose your API key in client-side code — use NEXT_PUBLIC_ env var only for the public key)
