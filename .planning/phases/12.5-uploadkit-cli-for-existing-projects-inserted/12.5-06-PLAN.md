<task type="auto" tdd="true">
  <name>Task 2: Codemod-based insertion + per-component templates</name>
  <files>packages/uploadkit-cli/src/add/insert-component.ts, packages/uploadkit-cli/src/add/templates/*.tsx.tpl</files>
  <behavior>
    insertComponent(filePath, alias, session):
    1. Read target file
    2. If marker block `// uploadkit:start — {alias}` already present →
       log "already present" and return {skipped: true}
    3. Back up file via session.save
    4. Use addImport to add `import { X } from "@uploadkitdev/react"`
    5. Load template for alias; wrap in markers
    6. Insert usage snippet:
       - If file has a default-exported function returning JSX → insert
         the snippet as FIRST child of the top-level returned JSX element
       - Else → append after the last import, as a commented reference
         block (rare fallback)
    7. Write file
    Six templates (one per alias) contain ONLY the JSX usage snippet
    (imports are added separately by addImport so they merge cleanly).
    Snippets must be copy-pasted into the first top-level element via a
    magicast + jscodeshift helper (per D-03 fallback allowed).

    Tests:
    - Given a fixture page.tsx (a Next.js page with `export default
      function Page() { return &lt;main&gt;&lt;/main&gt; }`), running
      insertComponent adds the import and drops a `<UploadDropzone />`
      inside `<main>`
    - Re-running leaves file unchanged
    - File still typechecks after insertion (spawn tsc against fixture)
  </behavior>
  <action>
    For the JSX insertion step: if magicast's JSX tree API can't express
    "insert as first child of returned JSX", drop to jscodeshift with a
    targeted transform. Wrap in a shared helper so the fallback is
    contained.
    Templates are intentionally tiny — one-line snippets. The docs guide
    (plan 10) shows full configured examples.
  </action>
