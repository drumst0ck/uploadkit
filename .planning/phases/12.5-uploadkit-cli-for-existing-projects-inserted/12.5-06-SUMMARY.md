# Phase 12.5 Plan 06: `add` Subcommand Summary

Shipped `npx uploadkit add <component>` for React frameworks. The command validates a 6-entry alias catalog (D-09), detects the host framework, prompts for a target .tsx/.jsx file (or honours `--target`), and splices a marker-bounded JSX snippet as the first child of the page's returned element — with the matching named import added via magicast. Second invocation against the same file+alias is a no-op. SvelteKit / Pages Router / unknown frameworks are refused politely. All 102 CLI tests pass (including 19 new ones for plan 06).

| Alias      | SDK export          | Source file                             |
| ---------- | ------------------- | --------------------------------------- |
| `dropzone` | `UploadDropzone`    | `components/upload-dropzone.tsx`        |
| `button`   | `UploadButton`      | `components/upload-button.tsx`          |

A spot-check (`grep '^export' packages/react/src/index.ts | grep -E '(UploadDropzone|UploadButton|UploadModal|UploadGalleryGrid|FileList|UploadProgressBar)'`) returns all six.

## What was built

### Task 1 — Catalog + framework gating + target resolver

`src/add/catalog.ts` — tiny, pure data module. Freezes the 6 alias → {import, pkg} map with `as const satisfies Record<string, ComponentSpec>` so mis-typed entries surface at compile time. Exposes `COMPONENT_ALIASES` (array), `isComponentAlias` (type guard), and `ComponentAlias` (type).

`src/add/prompt-target.ts` — wraps `fast-glob@latest` with per-framework candidate patterns:

| Framework    | Globs                                                                  |
| ------------ | ---------------------------------------------------------------------- |
| `next-app`   | `app/**/page.{tsx,jsx}`, `src/app/**/page.{tsx,jsx}`                   |
| `remix`      | `app/routes/**/*.{tsx,jsx}`                                            |
| `vite-react` | `src/**/*.{tsx,jsx}` (with `main.tsx`, `main.jsx` ignored)             |

Ignore list strips `node_modules`, `dist`, `.next`, `.svelte-kit`, `*.d.ts`, and test files. `--target` shortcircuits the glob, validates existence + `.tsx`/`.jsx` extension, and returns absolute. Non-interactive mode (`--yes` or no candidates) skips `@clack/prompts` entirely — picks `candidates.sort()[0]` or throws a clear error when empty.

`src/commands/add.ts` — orchestrator. Flow:

1. Validate positional alias → `printCatalog()` + exit 1 if unknown/missing.
2. `detectFramework(cwd)` →
   - `sveltekit` → `"\`add\` supports React frameworks only — SvelteKit components aren't shipped yet"` (exit 1)
   - `next-pages` → Pages Router refused (exit 1)
   - `unknown` → prints detection reason (exit 1)
   - supported → continue
3. `promptForTargetFile()` → absolute path.
4. Open backup session; call `insertComponent`.
5. On skipped=true → yellow "already present" (exit 0). On success → finalise session + green "Added <alias> to <path>".

### Task 2 — `insertComponent` + per-component templates

`src/add/insert-component.ts` — the real mutation. Behaviour:

1. Read the target file.
2. If `uploadkit:start — <alias>` is already in the source → `{skipped:true}` (no backup taken, no write).
3. `session.save(filePath)` — byte-for-byte backup before touching the file.
4. `addImport` (from 12.5-03) merges `import { <ImportName> } from "@uploadkitdev/react"`. Magicast handles deduplication — re-adding an already-imported specifier is a no-op.
5. `findInsertionSite(source)` — regex locates `export default function … (…)` or `export default (…) => …`, then scans for the next `return (<Elem` or `return <Elem`, then walks forward to the first `>` (bailing on `/>`). Returns `{ insertAt, indent }`. Indent is inferred from the next non-empty line inside the parent, falling back to `opening-tag-column + 2`.
6. If a site is found → splice `\n<indent>{/* uploadkit:start — <alias> */}\n<indent><snippet>\n<indent>{/* uploadkit:end — <alias> */}` immediately after the opening `>`. Existing children are preserved in place.
7. Fallback (no site, self-closing return, etc.) → append a `/* uploadkit:start — <alias> */ … /* uploadkit:end — <alias> */` block comment after the last `import` statement. Rare but guarantees the import isn't orphaned.

Six one-line `.tpl` snippets (`dropzone.tsx.tpl`, `button.tsx.tpl`, …). Intentionally minimal — the docs guide (plan 10) carries the configured examples. Templates are resolved at runtime from either `src/add/templates` (vitest) or `dist/add-templates` (built bin); tsup's `onSuccess` now copies BOTH template dirs (init unchanged, add new).

## Magicast vs jscodeshift notes

Tried jscodeshift for the JSX insertion step first, then abandoned it. The actual tree manipulation needed here is "insert as first child of the returned element" — a single, shallow, regex-expressible mutation. Bringing in jscodeshift would have:

- Added ~2 MB to the bundled CLI.
- Required its transform-runner harness (it's a codemod CLI, not just a library).
- Offered no idempotency guarantees magicast markers don't already give us.

Magicast at 0.5.2 still has no practical JSX-children API (same limitation surfaced in 12.5-03 `wrapChildrenOf`). So: addImport stays on magicast, JSX splicing is a marker-bounded regex edit, and both forms share the `uploadkit:start/end` idempotency key. The resulting diff is minimal, Prettier-safe, and reversible.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 — Missing critical functionality] Separate dist directory for add templates**

- **Found during:** Task 2, when sketching `loadTemplate`. `initNextApp` already owns `dist/templates` (12.5-04). Merging add templates into the same dir would have required renaming files (conflicts possible) or refactoring `initNextApp`'s loader — both out of scope for this plan.
- **Fix:** Extended `tsup.config.ts` `onSuccess` to ALSO copy `src/add/templates → dist/add-templates`. `insert-component.ts` resolves from two candidate dirs (src-side + dist-side) in order.
- **Files modified:** `packages/uploadkit-cli/tsup.config.ts`
- **Commit:** `6a31181`

### Inside Claude's discretion

**2. Per-component marker variant.** The plan's `<behavior>` says "Insertion includes required import AND usage bounded by markers". It doesn't specify whether the marker is shared (one per file) or per-component. `add` needs per-component so a user can drop a dropzone AND a gallery into the same page; hence `uploadkit:start — <alias>`. The layout-level `uploadkit:start` from 12.5-03/04 still exists for init — no conflict since `add` targets user pages, not layouts.

**3. `--yes` non-interactive behaviour picks first candidate deterministically.** The plan didn't mandate how `--yes` interacts with the select prompt. Deterministic first-sorted candidate keeps CI smoke (plan 09) reproducible without needing a test-only escape hatch.

**4. No cropper alias.** D-09 itself authorises dropping the cropper ("No cropper component exists in SDK at planning time; drop from the initial list"). Mentioned here so the SUMMARY explicitly acknowledges it.
