# Phase 12.5 Plan 03: Codemod Primitives + Backup Pipeline Summary

Delivered the shared primitives every insertion plan in Wave 3 depends on: AST-based import insertion via magicast, a pure-string env-file merger, a marker-bounded JSX wrapper, and a deferred-write backup session with a manifest format the `restore` subcommand (plan 07) can replay. Two test files, 25 tests specific to this plan (58 total in the CLI suite), all green.

## What was built

### Task 1 — Markers, imports codemod, env merge

**`codemods/markers.ts`** exports the two canonical line-comment strings from D-06:

```ts
export const MARKER_START = '// uploadkit:start';
export const MARKER_END   = '// uploadkit:end';
```

Plus three helpers:
- `hasMarkers(source)` — detects a complete pair (start followed by end on a later line). False for an unclosed start.
- `wrapWithMarkers(block)` — returns `MARKER_START — do not edit this block manually\n<block>\nMARKER_END`. The human-readable suffix on the start marker is inspired by shadcn's generated blocks; it survives Prettier reflow because the `// uploadkit:start` prefix is preserved.
- `stripMarkerBlock(source)` — returns `{ stripped, block }`. Used by restore logic in plan 07; included here so the shape is locked early.

**`codemods/imports.ts`** exports `addImport(source, { from, specifiers })`:

```ts
const mod = parseModule(source);
const existingLocals = new Set<string>();
for (const item of mod.imports.$items) {
  if (item.from === spec.from) existingLocals.add(item.local);
}
for (const name of spec.specifiers) {
  if (existingLocals.has(name)) continue;
  mod.imports.$append({ from: spec.from, imported: name, local: name });
}
return generateCode(mod).code;
```

Idempotency is by `local` name under the same `from` module — re-running `init` is a no-op on already-imported specifiers. The helper short-circuits and returns the original source string (not the regenerated code) when nothing changes, which keeps file mtimes untouched on re-runs and plays nicely with file watchers in user dev servers.

**`codemods/env.ts`** exports `mergeEnv(path, entries)`. Pure string manipulation — we do not pull dotenv. The parser splits on `/\r?\n/`, ignores blank lines and lines starting with `#`, and builds a `Set` of existing keys by splitting at the first `=`. Missing keys are appended as `KEY=VALUE\n` after ensuring the existing content ends with a newline. Commented-out keys (`# UPLOADKIT_API_KEY=…`) are treated as absent so the real key gets appended without disturbing the comment. Returns the list of keys appended (empty array if every entry already existed).

### Task 2 — JSX wrap codemod + backup pipeline

**`codemods/jsx-wrap.ts`** exports `wrapChildrenOf(source, { parentTag, wrapperOpen, wrapperClose })`. Magicast at 0.5.2 has no practical API for mutating JSX children (its proxy model targets imports and top-level declarations). Per D-03 we fall back to a regex-bounded string edit, with marker comments providing the idempotency guarantee magicast would normally give us:

1. Find the first `<parentTag …>` opening tag and its matching `</parentTag>`.
2. If the enclosed content already contains `uploadkit:start`, return source unchanged.
3. Otherwise splice `MARKER_START → wrapperOpen → existing children → wrapperClose → MARKER_END` inside the parent, preserving the existing children exactly.

The helper detects the indentation used by the first non-blank inner line and aligns the inserted wrapper to match. Sibling providers (`<ThemeProvider>`, `<AuthProvider>`, …) are never touched — we insert INSIDE them, satisfying phase risk #3. Throws clear errors on a missing parent tag or a self-closing variant.

**`backup/types.ts`** pins the on-disk manifest shape:

```ts
interface BackupManifest {
  timestamp: string;   // filesystem-safe ISO, e.g. 2026-04-15T14-32-07Z
  root: string;        // absolute repo root
  entries: Array<{
    action: 'modify' | 'create';
    relativePath: string;          // relative to root, OS-native separators
    backupPath?: string;           // relative to session dir, only for 'modify'
  }>;
}
```

**`backup/backup.ts`** exports `createBackupSession(root, { timestamp? })`:
- `save(absPath)` — validates the path is inside root, copies via `fs.cp` preserving the relative path inside the session dir, records a `modify` entry.
- `recordCreate(absPath)` — records a `create` entry without touching the filesystem (used for the route handler file / client module that `init` will create).
- `finalize()` — writes `manifest.json` into the session dir, then ensures `.uploadkit-backup/` is in `<root>/.gitignore`. Idempotent: a line-by-line match prevents duplicate entries when `init` is run repeatedly.

`save()`/`recordCreate()` validate paths are inside `root` via `path.relative` — a path escaping the root (`../../etc/passwd`) throws synchronously. This is defense-in-depth: plans 04–06 should only pass project-local paths, but the session enforces the invariant.

**`backup/restore-manifest.ts`** exports `readManifests(root)` for plan 07: walks `.uploadkit-backup/*/manifest.json`, parses each, sorts by timestamp descending, silently skips malformed manifests (better than failing restore), and returns `[]` when the backup dir doesn't exist.

`magicast@0.5.2` — unchanged from 12.5-01. No bump needed; the `parseModule` + `imports.$append` API we use is present and stable at this version.

## Known limitations of wrapChildrenOf (as called out in `<output>`)

- Assumes ONE `<parentTag>` per file. Next.js layouts have exactly one `<body>`; Remix `app/root.tsx` has one `<body>`. Multiple instances would match the first only.
- Cannot handle a parent tag whose opening is produced by a dynamic expression (e.g. `<Component as="body">` or a templated tag). The opening must be literal JSX.
- Self-closing `<parentTag />` throws — there is nothing to wrap. Callers should detect this upstream and refuse with a user-friendly error.
- Cannot preserve exotic prettier quirks (e.g. attributes split across many lines with unusual indentation). The inserted block uses inferred indentation from the first inner child.

These are intentional trade-offs for a reliable, regex-bounded edit with marker-based rollback. Plans 04/05 must snapshot-test against their framework's actual generated-layout shapes.

## Exact marker strings used (as called out in `<output>`)

```
// uploadkit:start — do not edit this block manually
...inserted lines...
// uploadkit:end
```

The `— do not edit this block manually` suffix is only on the start marker (end is bare). Detection (`hasMarkers`) matches the raw `// uploadkit:start` prefix, so the suffix text can evolve in future plans without breaking idempotency.

## Deviations from Plan

None of substance. Three choices inside Claude's discretion:

1. **wrapChildrenOf is regex-based, not magicast-based.** The plan's `<behavior>` explicitly allows this fallback ("if magicast can't express it, fall back to regex-bounded string edit marked clearly as fallback"). I added a JSDoc "Known limitations" block and an inline comment pointing at D-03 so future maintainers don't wonder why.
2. **Filesystem-safe timestamp format** (`2026-04-15T14-32-07Z`, no colons). Plan says "ISO 8601" — the replaced-colon variant is still unambiguous ISO-ish and works as a directory name on Windows. Windows CI is out of scope (D-10) but local Windows dev isn't, and this is free.
3. **Backup session validates paths are inside root.** Not in the plan, but Rule 2 (critical functionality for correctness/security) — a caller passing an absolute path outside the repo would silently write backup copies to unexpected locations. A synchronous throw catches the bug immediately. Zero cost for correct callers.

None of these change the behavioral contract of the `<done>` criteria.

- **Plan 12.5-05 (SvelteKit/Remix/Vite init)**: Remix uses the same `wrapChildrenOf('body', ...)` pattern against `app/root.tsx`. SvelteKit skips JSX wrap entirely (D-05: write `src/lib/uploadkit.ts` instead — use `mergeEnv` + `recordCreate`). Vite uses `wrapChildrenOf` against whatever root element `src/main.tsx`/`App.tsx` renders — expect to parameterize `parentTag` = `App` or similar.
- **Plan 12.5-06 (add)**: Uses `addImport` for the component import and `wrapWithMarkers(componentJsx)` for the inserted block. Each component gets its own marker pair; `hasMarkers` alone is insufficient for per-component idempotency — the `add` command should grep for the specific component name before inserting.
- **Plan 12.5-07 (restore)**: `readManifests(root)` lists sessions; iterate `entries` in REVERSE to undo (delete created files first, then copy modifies back). The `backupPath` field is relative to the session dir — resolve via `join(session.dir, entry.backupPath)`.
