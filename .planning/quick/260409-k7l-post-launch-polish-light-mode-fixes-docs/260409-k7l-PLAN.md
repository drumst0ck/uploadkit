**B. apps/docs/content/docs/sdk/react/before-upload.mdx (NEW)**
```mdx
---
title: Before Upload Hook
description: Transform, filter, or validate files client-side before they are uploaded.
---
```
Content:
- `onBeforeUploadBegin(files: File[]): File[] | Promise<File[]>` — called after selection, before presigned URL request
- Return empty array to cancel the upload entirely
- Example 1: rename files with a timestamp prefix
  ```ts
  onBeforeUploadBegin={(files) =>
    files.map(f => new File([f], `${Date.now()}-${f.name}`, { type: f.type }))
  }
  ```
- Example 2: filter to only images
  ```ts
  onBeforeUploadBegin={(files) => files.filter(f => f.type.startsWith('image/'))}
  ```
- Example 3: client-side image resize using canvas (show async pattern)
- Callout: "Server-side validation in your FileRouter still runs — onBeforeUploadBegin is a convenience hook, not a security boundary"

**D. apps/docs/content/docs/sdk/react/progress.mdx (NEW)**
```mdx
---
title: Upload Progress
description: Control progress reporting granularity for custom progress UIs.
---
```
Content:
- `uploadProgressGranularity: 'coarse' | 'fine' | 'all'`
- `coarse` (default): progress reported at 0%, 25%, 50%, 75%, 100%
- `fine`: every ~5% increment
- `all`: every XHR progress event (highest frequency, use for smooth progress bars)
- Code example with `onUploadProgress` callback showing a custom progress bar
  ```tsx
  <UploadButton
    route="imageUploader"
    uploadProgressGranularity="all"
    onUploadProgress={({ progress }) => setProgress(progress)}
  />
  ```
- Callout: "Use 'coarse' (default) unless you're rendering a custom animated progress bar — excessive re-renders can degrade upload performance"

**E. apps/docs/content/docs/sdk/next/ssr-plugin.mdx (NEW)**
```mdx
---
title: NextSSRPlugin
description: Prevent hydration mismatches and loading flashes in Next.js App Router.
---
```
Content:
- Problem: without the plugin, components that need the file router config trigger a loading flash on first render (SSR/client mismatch)
- Solution: `NextSSRPlugin` + `extractRouterConfig` in root layout.tsx
- Setup code:
  ```tsx
  // app/layout.tsx
  import { NextSSRPlugin } from '@uploadkit/next/server';
  import { extractRouterConfig } from '@uploadkit/next';
  import { ourFileRouter } from '@/app/api/uploadkit/core';

  export default function RootLayout({ children }) {
    return (
      <html>
        <body>
          <NextSSRPlugin routerConfig={extractRouterConfig(ourFileRouter)} />
          {children}
        </body>
      </html>
    );
  }
  ```
- Callout: "NextSSRPlugin is a Server Component. Import ourFileRouter from your server-side route handler — never from a client component."

**F. apps/docs/content/docs/sdk/next/backend-adapters.mdx (NEW)**
```mdx
---
title: Backend Adapters
description: Use UploadKit with Express, Fastify, or Hono — same FileRouter, different runtime.
---
```
Content:
- Intro: adapters expose the same `createUploadKitHandler` but wrap it for each framework's request/response interface
- Express example:
  ```ts
  import { createExpressHandler } from '@uploadkit/next/adapters/express';
  app.use('/api/uploadkit', createExpressHandler({ router: ourFileRouter, config: { ... } }));
  ```
- Fastify example:
  ```ts
  import { createFastifyHandler } from '@uploadkit/next/adapters/fastify';
  fastify.all('/api/uploadkit/*', createFastifyHandler({ router: ourFileRouter, config: { ... } }));
  ```
- Hono example:
  ```ts
  import { createHonoHandler } from '@uploadkit/next/adapters/hono';
  app.all('/api/uploadkit/*', createHonoHandler({ router: ourFileRouter, config: { ... } }));
  ```
- Note: the `endpoint` prop on React components points to your adapter URL, same as with Next.js
- Callout: "All adapters share the same FileRouter definition — swap the adapter without changing your frontend code"
