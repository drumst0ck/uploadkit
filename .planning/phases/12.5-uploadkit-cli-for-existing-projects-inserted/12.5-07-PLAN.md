Implement `uploadkit restore` to roll back changes made by `init` or
`add`. Reads backup manifests, prompts the user to pick one (or honors
`--timestamp`), then reverses every recorded action: copies backed-up
files back in place, deletes files the CLI created.

<interfaces>
Manifest shape (from plan 03):
```typescript
interface BackupManifest {
  timestamp: string;
  root: string;
  entries: Array<{
    action: "modify" | "create";
    relativePath: string;
    backupPath?: string;
  }>;
}
```

Restore algorithm:
- For each entry in reverse order:
  - action "modify" → copy `backupPath` back to `root/relativePath`
  - action "create" → delete `root/relativePath` (ignore ENOENT)
- Print summary table

If a restored file has been further modified by the user since backup,
prompt for confirmation before overwrite (unless `--yes`).
</interfaces>

    - `readManifests(cwd)` → if empty, print "No backup sessions found"

    - If `--timestamp` flag → filter to exact match or exit 1
    - Else → pickSession prompt returns chosen manifest
    - Delegate to applyRestore; print per-file status

    apply.ts:
    - applyRestore(manifest, { yes }): iterates entries reverse
    - For modify: reads current file bytes, compares sha256 to backup's
      sha256 (compute from backupPath); if drift detected AND !yes →
      prompt confirm; else copy backup over current
    - For create: fs.rm(path, { force: true })
    - Returns { restored: string[], deleted: string[], skipped: string[] }

    prompt.ts: select listing `${timestamp} (${entries.length} files)`.

    Tests:
    - Seed tmpdir with a manifest + backup dir + a created file;
      applyRestore removes the created file and restores the modified
      one
    - Running apply twice is safe (second call skips already-restored)
    - `--yes` bypasses the drift prompt
  </behavior>
