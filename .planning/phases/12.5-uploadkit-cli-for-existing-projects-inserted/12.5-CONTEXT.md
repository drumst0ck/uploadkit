# Phase 12.5 ‚Äî Locked Decisions

This document records the locked architectural decisions for Phase 12.5
(`npx uploadkit init` + `npx uploadkit add` for existing projects). All
sub-plans MUST honor these decisions verbatim.

## D-01 ‚Äî Package name: `uploadkit` (unscoped)

The CLI publishes as the unscoped npm package **`uploadkit`** so that
`npx uploadkit init` / `npx uploadkit add dropzone` read naturally, mirroring
the `shadcn` / `vercel` ergonomics. The package lives at
`packages/uploadkit-cli/` inside the monorepo. Binary name: `uploadkit`.

Rationale: `npx @uploadkitdev/cli init` is long and unfriendly; developers
searching "uploadkit cli" expect the short form. The scoped packages
(`@uploadkitdev/react`, `@uploadkitdev/next`, etc.) remain the runtime SDK
surface.

## D-02 ‚Äî Reuse Phase 12 tooling, NEW package (no fork)

Phase 12 shipped `packages/create-uploadkit-app/` with `@clack/prompts`,
`mri`, `execa`, `picocolors`, `tsup`, and `vitest`. Phase 12.5 creates a
**new package** (`packages/uploadkit-cli/`) that reuses the *same tools*
but ships a different binary with a different surface. We do NOT extend
`create-uploadkit-app` ‚Äî its scope is greenfield scaffolding.

Shared code (PM detection, prompt helpers) is duplicated at copy level
for now; if a third CLI package appears we extract to `packages/cli-utils/`.
Keeping two small packages independent beats premature abstraction.

## D-03 ‚Äî AST tool: `magicast` primary, `jscodeshift` fallback

**`magicast`** is the primary tool for JS/TS edits (adding imports,
inserting JSX children, writing env vars, wrapping root providers). It is
purpose-built for config-mutation use cases (Nuxt, Vite, Next.js configs
and layouts) and produces minimal diffs that prettier/biome leave alone.

**`jscodeshift`** is available as a fallback for edits magicast cannot
express (complex JSX tree manipulation). Prefer magicast when possible.

For framework-specific files:
- `.svelte` files: parse with `svelte/compiler` `parse()` + string edits
  bounded by marker comments (no full SFC rewrite).
- `.vue` files: **out of scope** this phase (no Vue template today).

## D-04 ‚Äî Framework support matrix

| Framework | Detection | `init` supported | `add` supported |
|-----------|-----------|-------|-------|
| Next.js App Router | `next` in deps + `app/layout.*` | yes | yes |
| Next.js Pages Router | `next` + `pages/_app.*` (no `app/`) | refuse with message | no |
| SvelteKit | `@sveltejs/kit` in deps | yes | no (React-only components) |
| Remix / React Router 7 | `@remix-run/*` or `@react-router/dev` | yes | yes |
| Vite + React | `vite` + `react` in deps | yes (client-only warning) | yes |
| Anything else | fallthrough | refuse with message | refuse |

Detection reads `package.json` dependencies FIRST, then inspects config
files (`next.config.*`, `svelte.config.*`, `react-router.config.*`,
`vite.config.*`) as a tiebreaker.

## D-05 ‚Äî Insertion targets per framework

- **Next.js App Router**:
  - Route handler: `app/api/uploadkit/[...uploadkit]/route.ts`
  - Provider mount: wrap `<body>` children in `app/layout.tsx` with
    `<UploadKitProvider>`
- **SvelteKit**:
  - Route handler: `src/routes/api/uploadkit/[...uploadkit]/+server.ts`
  - No React provider; write a typed `uploadkit` client instance in
    `src/lib/uploadkit.ts` instead
- **Remix / RR7**:
  - Route: `app/routes/api.uploadkit.$.tsx` with `action`/`loader`
  - Provider mount: wrap children in `app/root.tsx`
- **Vite + React**:
  - No server; provider mounted in `src/main.tsx` or `src/App.tsx`
  - Warn user: "Vite has no server ‚Äî uploads will use BYOS/presigned URLs"

## D-06 ‚Äî Idempotency markers

Every CLI-inserted block is bounded by marker comments:
```ts
// uploadkit:start ‚Äî do not edit this block manually
...inserted code...
// uploadkit:end
```
`init` detects prior install by grepping for `uploadkit:start` in the
target files. If present, emit "already configured" and skip. `add`
uses the same markers per component insertion (one block per component).

## D-07 ‚Äî Backup strategy

Before ANY file modification:
1. Create `.uploadkit-backup/<ISO-timestamp>/` at repo root.
2. Copy original file with its relative path preserved.
3. Write a `manifest.json` listing each backup with action + target.
4. Add `.uploadkit-backup/` to `.gitignore` automatically.

`uploadkit restore` lists timestamps, prompts for one, and replays the
manifest in reverse (copying backed-up files back, deleting files the
CLI created).

## D-08 ‚Äî Subcommand surface

```
uploadkit init [--yes] [--skip-install]
uploadkit add <component> [--target <path>] [--yes]
uploadkit restore [--timestamp <iso>]
uploadkit --version
uploadkit --help
```

Flags parsed via `mri` (same as Phase 12). Interactive prompts via
`@clack/prompts`.

## D-09 ‚Äî Component list for `add`

Verified against `packages/react/src/components/`. The `add` subcommand
ships with these canonical aliases mapped to SDK exports:

- `dropzone` ‚Üí `UploadDropzone` (from `upload-dropzone.tsx`)
- `button` ‚Üí `UploadButton` (from `upload-button.tsx`)
- `modal` ‚Üí `UploadModal` (from `upload-modal.tsx`)
- `gallery` ‚Üí `UploadGalleryGrid` (from `upload-gallery-grid.tsx`)
- `queue` ‚Üí `FileList` (from `file-list.tsx` ‚Äî used as upload queue)
- `progress` ‚Üí `UploadProgressBar` (from `upload-progress-bar.tsx`)

No cropper component exists in SDK at planning time; drop from the initial
list (can be added later when the SDK ships one). All six map 1:1 to
files present in `packages/react/src/components/`.

## D-10 ‚Äî Explicit scope cuts (out of scope this phase)

- Next.js Pages Router (App Router required)
- Vue / Nuxt / Angular / SolidStart / Astro
- `add` for SvelteKit (Svelte component variants don't exist yet)
- Monorepo-aware installs (assume single `package.json` at CWD)
- Custom build setups (webpack, esbuild-alone, rspack) not matching
  the detection matrix
- Windows smoke CI (deferred per Phase 12 follow-up ‚Äî Linux only)
- Templates that are already covered by `create-uploadkit-app` (this
  CLI is for EXISTING projects, not scaffolding new ones)

## D-11 ‚Äî Publish and CI gating

- Package published as `uploadkit@0.1.0` via Changesets + the existing
  `release.yml` workflow.
- Reuse Phase 12's smoke-test pattern: a matrix of fixture apps (one per
  supported framework) is scaffolded in CI, `uploadkit init` runs against
  each, the resulting project must `typecheck + build` successfully.
- Smoke gate MUST pass before publish (blocking).

## D-12 ‚Äî Docs placement

A new MDX guide at `apps/docs/content/docs/guides/cli-existing-projects.mdx`.
If `apps/docs/content/docs/guides/cli.mdx` exists (Phase 12 output), add
a sibling page and cross-link. Both the `init` and `add` flows get
framework-specific tabs.
