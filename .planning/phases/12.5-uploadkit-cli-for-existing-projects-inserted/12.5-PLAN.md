# Phase 12.5 — `uploadkit` CLI for existing projects

## Goal (outcome)

A developer with an existing Next.js / SvelteKit / Remix / Vite+React
project runs `npx uploadkit init`, answers a few prompts, and has
UploadKit fully wired in under 60 seconds — deps installed, route handler
created, provider mounted in their root layout, env vars scaffolded, and
everything backed up in case they want to roll back. `npx uploadkit add
<component>` then drops individual React components shadcn-style.

## In scope

- New package `packages/uploadkit-cli/` publishing as npm `uploadkit`
- Subcommands: `init`, `add`, `restore`, `--version`, `--help`
- Framework detection: Next.js App Router, SvelteKit, Remix/RR7, Vite+React
- AST-based code modification via `magicast` with marker-comment bounded
  insertions for idempotency
- Backup-and-restore pipeline (`.uploadkit-backup/<timestamp>/`)
- Fixture-driven tests + CI smoke matrix
- Docs page at `/docs/guides/cli-existing-projects`
- Changesets-gated npm publish

## Out of scope

See `12.5-CONTEXT.md` D-10. Highlights: Pages Router, Vue/Nuxt/Angular,
monorepo-aware installs, Windows smoke CI, Svelte component variants for
`add`, greenfield scaffolding (Phase 12 already covers it).

## Risks

1. **AST edits drift after user formatting**. Prettier or biome may
   reflow our marker-bounded blocks. Mitigation: keep inserted code
   idiomatic on ONE line where possible; use marker comments that are
   still recognizable after reformat (line-comment form, not block).
2. **Framework detection false positives**. A Vite repo that also has
   `next` as a dev dep could be misclassified. Mitigation: prioritize
   framework config files (`next.config.*` wins over deps alone); prompt
   user to confirm the detected framework before mutating.
3. **Provider wrapping breaks user JSX**. If the user's `app/layout.tsx`
   has an existing provider chain, we must insert INSIDE, not replace.
   Mitigation: only wrap `<body>`'s direct children and preserve all
   existing wrappers. Snapshot-test against hand-crafted realistic
   layouts.
4. **Idempotency marker loss**. If the user deletes our markers, `init`
   would re-insert and create duplicates. Mitigation: additionally check
   for `@uploadkitdev/react` import presence before inserting provider;
   bail with a clear message if conflicts detected.
5. **npm package-name squat**. `uploadkit` may be taken or reserved.
   Mitigation: claim in plan 10; fall back to `@uploadkitdev/cli` with a
   `uploadkit` alias binary published separately if the name is truly
   unavailable. Plan 10 includes a check-and-confirm step.
6. **`.env.local` merge conflicts**. Existing keys could collide.
   Mitigation: read existing file, append missing keys only, never
   overwrite existing values; print a diff summary.

## Success criteria (phase-level)

1. `npx uploadkit init` succeeds in <60s on a fresh Next.js App Router
   project (measured in CI smoke).
2. Running `init` a second time prints "already configured" and makes
   zero file changes (idempotency asserted by test).
3. `npx uploadkit add dropzone` inserts a working component in a
   user-selected page, producing a file that typechecks.
4. `.uploadkit-backup/` is created and `uploadkit restore` rolls back
   all `init` changes verifiably (fixture test).
5. Package `uploadkit@0.1.0` is published to npm via release.yml after
   smoke gate passes.
6. Docs page live at `/docs/guides/cli-existing-projects` with both
   flows documented per framework.
7. All sub-plan `<verify>` commands pass on Linux.

## Verification strategy

- Per-plan `<verify>` runs Vitest on the affected module.
- Phase-level verification: CI smoke matrix (plan 09) runs `init` end-to-end
  against one fixture app per framework.
- Manual checkpoint (plan 10): maintainer confirms npm name availability
  and ships via Changesets.
