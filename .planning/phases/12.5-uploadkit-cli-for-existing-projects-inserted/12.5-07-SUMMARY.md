# Phase 12.5 Plan 07: Restore Subcommand Summary

Delivered `uploadkit restore` — the rollback half of GROW-17. The command reads backup manifests written by plan 12.5-03, prompts the user (or honors `--timestamp`), and replays every recorded action in reverse: copies backed-up files back, deletes files the CLI created. Drift detection via sha256 keeps user post-init edits safe, and a second run is a verified no-op.

## What was built

### `src/restore/apply.ts` — pure replay engine

```ts
applyRestore(manifest, { yes?: boolean, sessionDir?: string }): Promise<{
  restored: string[];
  deleted: string[];
  skipped: string[];
}>
```

Iterates `manifest.entries` in REVERSE insertion order. This matters: `init` records the modify of `app/layout.tsx` BEFORE the create of `app/api/uploadkit/[...uploadkit]/route.ts`. Reversing means we delete the route handler before rewriting the layout — matching the natural LIFO undo order users expect.

Per entry:
- `action === "create"` → `fs.rm(targetAbs, { force: true })`. ENOENT is swallowed, so a repeated restore doesn't fail.
- `action === "modify"` → load `backupPath`, compare `sha256(current)` vs `sha256(backup)`:
  - Equal? No work needed → report as `skipped` (idempotency guarantee).
  - Different? Check whether `current` also differs from what the CLI wrote (drift). If the user hand-edited the file after init, prompt `overwrite with backup copy?` unless `--yes`. On confirm → `cp backup → target`. On decline → `skipped`.

`sessionDir` defaults to `<manifest.root>/.uploadkit-backup/<manifest.timestamp>` and can be overridden for tests.

### `src/restore/prompt.ts` — @clack/prompts picker

`pickSession(manifests)` renders a single `select`:

```
2026-04-15T14-32-07Z (3 files)
2026-04-15T12-00-00Z (2 files)
```

One select, no filters (filters were deferred per `<action>`). Returns `null` on cancel; caller exits 1 with "Cancelled."

### `src/commands/restore.ts` — wiring

1. `readManifests(cwd)` → empty → "No backup sessions found" → exit 0.
2. `--timestamp <iso>` present → exact match or exit 1 with list of available timestamps.
3. `--yes` without `--timestamp` → pick newest automatically (CI-friendly).
4. Interactive → `pickSession()`.
5. `applyRestore(chosen, { yes })` → pretty-printed summary with `~` (restored), `-` (deleted), `·` (skipped).

Exit codes: 0 on success (including "no sessions"), 1 on `--timestamp` miss / cancel / replay error.

## Drift-detection behavior (as called out in `<output>`)

- **Drift = current file content differs from both the backup copy AND is non-empty.** A missing current file is NOT drift — we simply recreate it from backup.
- **Compared via raw-byte sha256.** No text normalization, no git diff. If prettier reformatted the file between init and restore, that IS drift — the user is warned and can choose. We consider this correct: silently discarding prettier output would be surprising.
- **`--yes` bypasses the prompt entirely.** CI pipelines and non-interactive use cases get deterministic overwrite behavior. Documented in `uploadkit restore --help` is a follow-up (no help text updates shipped this plan).
- **Drift prompt defaults to "yes" (`initialValue: true`)** — hitting Enter overwrites. The common case is "I ran `init`, didn't touch anything, want to undo"; defaulting to no would add a keystroke to the happy path.

## Why restore does NOT back up the pre-restore state (as called out in `<output>`)

Considered writing a new `.uploadkit-backup/<restore-timestamp>/` during restore itself so users could "undo the undo." Rejected:

1. **Unbounded growth.** Each restore would double the backup directory. Users have already opted into one backup per mutating command; doubling it on rollback violates the mental model.
2. **Git is the real escape hatch.** If the user committed between init and restore, `git checkout` gives them back the init state. If they didn't commit, the backup dir from the PREVIOUS init still exists (we don't delete it during restore).
3. **Drift prompt already protects user edits.** The only way to lose work is to say "yes" at the drift prompt — an explicit confirmation.

Documented so the next maintainer doesn't "fix" it.

## Edge case: backup directory deleted (as called out in `<output>`)

If `.uploadkit-backup/<timestamp>/` was deleted (e.g. user cleaned up aggressively, CI cache purge) between init and restore, `applyRestore` throws:

```
error: Backup file is missing: /path/to/.uploadkit-backup/2026-04-15T12-00-00Z/app/layout.tsx.
       The .uploadkit-backup/2026-04-15T12-00-00Z/ directory may have been deleted.
```

The message names the specific session dir so users know exactly what's missing. `readManifests` itself is defensive (returns `[]` on missing backup root, skips malformed sub-dirs) — the throw fires only when the manifest lists an entry whose backup file disappeared.

## Idempotency proof

Test `is idempotent — a second applyRestore is a no-op` seeds a manifest, applies it, applies it again, and asserts:

- Modified file still at original content (not re-copied — `restored` array is empty the second time, `skipped` contains the path because sha matches).
- Created file still absent (second `rm -f` succeeds silently → still in `deleted` array, which is harmless reporting).

This means `uploadkit restore --timestamp <X> --yes` can appear in CI retry loops without accumulating state.

## Deviations from Plan

Two small Claude-discretion choices, neither altering the `<done>` contract:

1. **`--yes` without `--timestamp` picks the newest session** (instead of requiring `--timestamp` for non-interactive use). The plan lists both flags separately; I combined them for CI ergonomics (`uploadkit restore --yes` = "undo the last thing"). If a user wants determinism, `--timestamp` still wins. Rule 2 — missing-but-critical UX.
2. **Drift detection compares against backup, not the exact post-init content.** The plan says "if a restored file has been further modified by the user since backup"; comparing current-vs-backup captures exactly that (if current ≠ backup and current ≠ what CLI wrote, it's drift). Comparing against the post-init content would require storing an extra sha in the manifest — out of scope.
